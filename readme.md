### Задача 1: Обеспечить разработку

#### Основное решение: Managed Service for GitLab в Yandex Cloud

Дополнительно: `Yandex Container Registry` + (опционально) `Yandex Lockbox` для внешнего хранения секретов.

`GitLab` закрывает почти все требования “из коробки”, оставаясь облачным сервисом в российском облаке, а `Yandex Cloud` даёт инфраструктурные сервисы вокруг (`registry`, секреты, сети, `VPN`).

---

#### Архитектура и взаимодействие компонентов

* **GitLab (Yandex Cloud Managed GitLab):**

  * Хранение исходного кода (Git).
  * Управление проектами/репозиториями (по одному репо на сервис).
  * GitLab CI/CD для сборки, тестов и деплоя.
  * Хранение CI‑переменных и секретов.
* **GitLab Runner:**

  * Часть раннеров — в Yandex Cloud (VM или k8s).
  * Часть раннеров — на собственных серверах (on‑prem), подключённых по HTTPS/VPN.
  * Раннеры используют собственные Docker‑образы для сборки.
* **Yandex Container Registry:**

  * Хранение Docker/OCI‑образов приложений и образов для CI.
  * Интеграция с GitLab CI через токены/robot‑аккаунты.
* **(Опционально) Yandex Lockbox:**

  * Хранение особо чувствительных секретов (доступы к прод‑БД, внешним API).
  * Доступ к секретам из CI через сервисные аккаунты Yandex Cloud.

---

#### Как решение закрывает требования

**Облачная система (российский облачный сервис)**

* Решение: GitLab как Managed Service в Yandex Cloud.
* Обоснование: размещение в российском облаке, SLA, резервирование, интеграция с другими сервисами Yandex Cloud.

**Система контроля версий Git, репозиторий на каждый сервис**

* **Решение:** GitLab Projects.
* **Принцип:**
  * Один микросервис = один GitLab‑проект/репозиторий.
  * Группы/подгруппы GitLab используются для логической группировки доменов (billing, auth, etc).
* **Обоснование:** классический Git‑workflow (feature‑ветки, MR, code review), гибкие права доступа.

**Запуск сборки по событию из системы контроля версий**

* **Решение:** GitLab CI pipelines, триггер по push, merge request, tag.
* **Принцип:**

  * В корне репозитория — `.gitlab-ci.yml`.
  * Определяем stages: `[build, test, deploy]`.
  * Для веток `feature/*` — автосборка и автотесты.
  * Для `develop/master` — дополнительные шаги (интеграционные тесты, деплой на `dev/stage`).
* **Обоснование:** нативная интеграция CI с Git, нет отдельного сервера для вебхуков.

**Запуск сборки по кнопке с указанием параметров**

* **Решение:** GitLab “Run pipeline” + CI variables.
* **Принцип:**

  * В `.gitlab-ci.yml` описываются переменные с `when: manual` или `trigger: manual`.
  * При запуске `pipeline` из UI можно задать параметры: `TARGET_ENV`, `FEATURE_FLAG`, `DEPLOY_VERSION` и т.д.
* **Обоснование:** удобно для ручных релизов, `hotfix`‑деплоев, прогонов специфических тестов.

**Возможность привязать настройки к каждой сборке**

* **Решение:** `GitLab CI variables` (project/group/instance level) + `variables` в `pipeline`.
* **Принцип:**

  * Общие настройки — на уровне группы (например, `URL registry`, общие токены).
  * Специфичные — на уровне проекта (доступ к конкретным БД/очередям).
  * Настройки конкретного запуска — через `“Run pipeline”` (override переменных).
* **Обоснование:** гибкая модель конфигурации, `audit trail` изменений.

**Шаблоны для различных конфигураций сборок**

* **Решение:** `GitLab CI templates` + `include`.
* **Принцип:**

  * Создаём общий репозиторий ci-templates с .gitlab-ci.yml‑фрагментами:
    * templates/docker-build.yml
    * templates/python-service.yml
    * templates/node-service.yml
* **В сервисах используем:**

  ```
   include:
   - project: 'ci/ci-templates'
     file: 'templates/python-service.yml'
  ```

* **Обоснование:** единые стандарты сборки, меньше дублирования, проще менять пайплайны массово.

**Безопасное хранение секретных данных**

* **Решение:** `GitLab CI/CD Variables` (`masked`, `protected`) + `Yandex Lockbox` (для критичных секретов).
* **Принцип:**

  * В GitLab:

    * Секреты как `masked` и `protected` переменные (пароли к тестовым БД, токены к `registry`).
    * Доступ к ним только в защищённых ветках/тегах.
  * В Yandex Lockbox:

    * Прод‑секреты (`prod‑DB`, внешние платёжные `API`).
    * В `CI` — получение секретов через `CLI/SDK Yandex Cloud` под сервисным аккаунтом.
* **Обоснование:** разделение уровней чувствительности, минимизация утечек через логи/артефакты.

**Несколько конфигураций для сборки из одного репозитория**

* **Решение:** несколько `jobs` и/или несколько `pipelines` с разными `rules/only/except`.
* **Принцип:**

  * В одном `.gitlab-ci.yml`:

    * `build_debug`, `build_release`, `build_arm`, `build_x86`.
    * `rules` по веткам, тегам, переменным.
  * Можно запускать нужную конфигурацию вручную, передавая переменную `BUILD_PROFILE`.
* **Обоснование:** один репозиторий — разные таргеты (образы, платформы, конфигурации).

**Кастомные шаги при сборке**

* **Решение:**  `GitLab CI jobs` с произвольными `shell`‑скриптами.
* **Принцип:**

  * Любой шаг описывается в `job`:

  ```
    custom_step:
    stage: build
    script:
        - ./scripts/generate-version.sh
        - ./scripts/custom-lint.sh
  ```

* **Обоснование:** нет ограничений на логику, можно интегрировать любые внутренние инструменты.

**Собственные Docker‑образы для сборки проектов**

* **Решение:**  `GitLab CI image`: + `Yandex Container Registry`.
* **Принцип:**

  * Создаём базовые образы для разных стеков (`python`, `node`, `java`, `go`) и пушим в `YCR`.
  * В `.gitlab-ci.yml`:

    ```
    image: cr.yandex/<registry-id>/ci/python:3.11
    ```

* **Обоснование:** воспроизводимая среда сборки, контроль версий образов, быстрые сборки.

**Возможность развернуть агентов сборки на собственных серверах**

* **Решение:**  GitLab Runner on‑prem.
* **Принцип:**

  * Устанавливаем `GitLab Runner` на свои сервера (bare metal или k8s).
  * Регистрируем раннеры в `Managed GitLab` (token).
  * Настраиваем теги раннеров (`onprem`, `gpu`, `secure`) и используем их в `jobs`:

    ```
    tags:
        - onprem
    ```

* **Обоснование:** можно использовать внутренние ресурсы, доступ к закрытым системам, соблюдение регуляторики.

**Параллельный запуск нескольких сборок**

* **Решение:** `GitLab CI concurrent jobs` + несколько раннеров.
* **Принцип:**

  * Настраиваем concurrent на раннерах и поднимаем нужное количество экземпляров.
  * `GitLab` сам распределяет `jobs` по доступным раннерам.
* **Обоснование:** масштабирование по горизонтали, сокращение времени ожидания сборок.

**Параллельный запуск тестов**

* **Решение:** `GitLab CI parallel`: и/или разбиение тестов на несколько `jobs`.
* **Принцип:**

  * Для большого набора тестов:

    ```
    tests:
        stage: test
        script: run_tests.sh
        parallel: 4
    ```

  * Либо отдельные jobs: unit_tests, integration_tests, e2e_tests в одной стадии.
* **Обоснование:** ускорение обратной связи для разработчиков, эффективное использование раннеров.

---

#### **Итоговое обоснование выбора**

* Единая платформа: `Git`, `CI/CD`, управление проектами и доступами — в одном продукте (GitLab), без “зоопарка” разрозненных систем.
* Соответствие требованиям: все перечисленные требования (триггеры, шаблоны, секреты, кастомные шаги, параллельность, свои образы, `on‑prem` агенты) — нативные возможности `GitLab`.
* Интеграция с российским облаком: `Yandex Cloud` даёт управляемый `GitLab`, `registry`, `секреты`, сети и возможность гибридной схемы (облако + свои сервера).
* Масштабируемость и стандартизация: шаблоны пайплайнов и собственные образы позволяют задать единые стандарты для всех микросервисов и масштабировать практики `DevOps` на всю компанию.

### Задача 2: Логи

#### Решение: Loki + Promtail + Grafana (в российском облаке)

**Облачная платформа:** VK Cloud / Yandex Cloud (оба поддерживают развёртывание Loki‑стека; выбираем по корпоративным требованиям).

**Компоненты решения:**

* Loki — центральное хранилище логов (аналог Elasticsearch, но оптимизирован под логи).
* Promtail — агент сбора логов на каждом хосте/ноде.
* Grafana — интерфейс поиска, фильтрации, дашбордов и сохранённых запросов.

Это решение хорошо подходит для микросервисной архитектуры, где сервисы пишут логи в stdout, а инфраструктура должна гарантировать доставку и удобный поиск

---

**Как компоненты взаимодействуют**

1. Приложения → stdout

   Микросервисы пишут структурированные логи (JSON или текст) в стандартный вывод.
2. Promtail (на каждом хосте / в DaemonSet в Kubernetes)

   * Читает stdout контейнеров через Docker/Kubernetes API.
   * Добавляет метаданные: имя сервиса, namespace, pod, labels.
   * Буферизует логи и гарантирует доставку (ретраи, очереди).
   * Отправляет логи в Loki по HTTP.
3. Loki (центральное хранилище)

   * Принимает логи от всех Promtail‑агентов.
   * Хранит логи в объектном хранилище (S3‑совместимое в VK Cloud / Yandex Cloud).
   * Индексирует только метаданные → дешёвое хранение, высокая скорость поиска.
4. Grafana

   * Подключается к Loki как datasource.
   * Предоставляет UI для поиска, фильтрации, сохранённых запросов.
   * Позволяет делиться ссылками на сохранённые поиски.
   * Управляет доступами разработчиков (RBAC).

---

**Как решение закрывает требования**

* Сбор логов в центральное хранилище со всех хостов

  * Promtail собирает логи с каждого узла.
  * Loki хранит их централизованно в облаке.
* Минимальные требования к приложениям — сбор из stdout

  * Promtail читает stdout контейнеров автоматически.
  * Приложения не должны уметь писать в файлы или отправлять логи вручную.
* Гарантированная доставка логов

  * Promtail имеет:

    * локальные очереди,
    * ретраи,
    * backpressure‑механику,
    * временное локальное хранилище при недоступности Loki.
  * Loki использует объектное хранилище → высокая надёжность.
* Поиск и фильтрация по записям логов

  * Grafana + Loki Query Language (LogQL):

    * фильтрация по сервису, уровню, pod, trace_id, тексту,
    * поиск по JSON‑полям,
    * агрегации (count, rate, histogram).
* Пользовательский интерфейс с доступом разработчиков

  * Grafana предоставляет удобный UI:

    * поиск логов,
    * сохранённые запросы,
    * дашборды,
    * RBAC (ограничение доступа по командам/сервисам).
* Возможность дать ссылку на сохранённый поиск

  * Grafana позволяет:

    * сохранять запросы,
    * делиться прямыми ссылками,
    * создавать папки с сохранёнными поисками для команд.

---

**Почему именно Loki‑стек**

1. Оптимизирован под микросервисы

   * Логи в stdout — стандарт для Kubernetes.
   * Loki хранит только метаданные → дешевле и быстрее, чем ELK.
2. Гарантированная доставка

   * Promtail надёжнее Filebeat/Fluentd в сценариях с нестабильной сетью.
3. Простота эксплуатации

   * Нет тяжёлых кластеров Elasticsearch.
   * Масштабируется горизонтально.
   * Хранение в S3‑совместимом объектном хранилище.
4. Отличный UI

   * Grafana — лучший интерфейс для логов, метрик и трейсинга.
5. Российские облака поддерживают Loki

   * VK Cloud Marketplace — готовые Loki‑стек решения.
   * Yandex Cloud — легко разворачивается через Managed Kubernetes + Object Storage.

---

**Итоговая архитектура**

```
[Микросервисы] → stdout
        ↓
[Promtail на каждом хосте / DaemonSet]
        ↓ (гарантированная доставка)
[Loki в облаке + Object Storage]
        ↓
[Grafana] → поиск, фильтрация, сохранённые запросы, доступы
```

### Задача 3. Мониторинг

Рекомендую стек на базе Prometheus (kube‑prometheus‑stack) для сбора метрик + node‑exporter, cAdvisor и kube‑state‑metrics для сбора хост‑ и контейнерных метрик, Grafana для UI, и VictoriaMetrics или Thanos для долговременного хранения и масштабирования. Это даёт сбор метрик со всех хостов и сервисов, сбор системных и сервисных метрик, гибкие запросы и настраиваемые дашборды.

---

**Ключевые компоненты и принципы взаимодействия**

* Prometheus (kube‑prometheus‑stack) — центральный сборщик метрик, использует ServiceMonitor/PodMonitor для автоматического обнаружения целей в Kubernetes; управляет правилами и алертами.
* node‑exporter — агент на каждом хосте для метрик CPU, RAM, HDD, network.
* cAdvisor / kubelet metrics — сбор метрик контейнеров (container_cpu_usage_seconds_total, memory, filesystem).
* kube‑state‑metrics — метрики состояния Kubernetes‑объектов (pod/replset/deployment status, requests/limits).
* Service‑specific metrics — приложения экспортируют метрики в формате Prometheus (/metrics, OpenMetrics / OpenTelemetry → Prometheus). Prometheus скрапит эти эндпоинты.
* Долговременное хранение и масштабирование: VictoriaMetrics или Thanos для ретенции (срок хранения данных), горизонтального масштабирования и глобального агрегирования.
* Grafana — UI для запросов (PromQL), агрегаций и создания панелей; управление доступом для разработчиков и создание шаблонных дашбордов.

---

**Как решение закрывает требования**

* Сбор со всех хостов: node‑exporter и cAdvisor разворачиваются как DaemonSet/агенты на всех нодах; Prometheus скрапит их.
* Системные ресурсы хостов: node‑exporter даёт CPU/RAM/disk/network метрики.
* Ресурсы на сервис: cAdvisor + kube‑state‑metrics + метрики контейнеров дают CPU/RAM/HDD/Network на уровне pod/container.
* Сервис‑специфичные метрики: приложения экспортируют свои метрики в /metrics; Prometheus их собирает.
* UI и агрегация: Grafana + PromQL; можно строить агрегированные панели (namespace, service, cluster).

---

| **Критерий**     | **Thanos**                                                                                    | **VictoriaMetrics**                                                                                                                  |
| ------------------------ | --------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| **Масштаб**       | Очень хорошо для глобального агрегирования; S3‑backed. | Высокая плотность хранения, оптимизирована по цене/производительности. |
| **Стоимость**   | Средняя–высокая (S3 + компоненты).                                   | Часто ниже при больших объёмах.                                                                            |
| **Сложность**   | Более сложен в развёртывании/операциях.                     | Проще в эксплуатации, есть k8s‑chart.                                                                         |
| **Лучше для**    | Мультикластерных сценариев и HA.                                    | Больших объёмов метрик с ограниченным бюджетом.                                             |
| **Интеграция** | Нативно с Prometheus.                                                                 | Нативно поддерживает remote_write; есть k8s‑stack.                                                           |

**Обоснование  выбора:**

Выбор Prometheus‑стека обоснован тем, что он полностью закрывает все требования мониторинга микросервисов при минимальной сложности и максимальной надёжности.  
Он собирает метрики со всех хостов и контейнеров, поддерживает сервис‑специфичные метрики через стандартный /metrics, масштабируется (Thanos/VictoriaMetrics), а Grafana даёт удобный интерфейс для запросов и дашбордов.
